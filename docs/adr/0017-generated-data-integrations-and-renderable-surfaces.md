# Generated data integrations and renderable surfaces

Autopilot will let a non-technical Company describe the data it wants to connect, understand, monitor, and operate without requiring a fixed catalog of hand-built integrations. The product therefore needs a durable boundary between generated integration code, governed data access, generated UI, and external effects. This ADR records the ratified decisions from the Dashboard, Mini-app, and Data Source domain-modeling grill and its integration authoring/runtime continuation (package tree, adapter contract, credentials, proxies, execution paths, ingress, lifecycle, and the proving slice); it remains proposed until the remaining Phase-2 scenarios and negative oracles are resolved.

## Decision

### Dashboard, Mini-app, and Surface

- A **Dashboard** is a read-oriented durable composition. Local filters, drill-down, and navigation are allowed; Company-data mutation occurs only through a named, authorized Integration Action with visible pending, success, failure, and approval state.
- A **Mini-app** is a real Company-owned, versioned zero-install source project with governed Data Query bindings, Integration Action bindings, private store grants, Runs, and optional Automation links. Its UI may contain `index.html`, normally imported local `*.jsx`/TypeScript modules, CSS, and template-provided JavaScript. Autopilot and authorized Humans edit the source files directly; qualification deterministically compiles each exact source revision into an immutable execution bundle. A Mini-app is not a Dashboard, deployed Project, hidden presentation-specific document model, or owner of a second server-action system.
- The first Mini-app slice has no Template Registry or Mini Store. Autopilot loads a built-in authoring Skill and its supporting reference files, then creates a concrete `.app` draft directly. A Slides Skill may teach the zero-install module layout and provide reference `slides.js`, `export.js`, `kit.jsx`, and example slide files for Autopilot to copy and specialize; these remain Skill resources rather than a new runtime domain.
- A versioned **Mini-app Template** is a later promotion of a proven repeated `.app` scaffold. Once real Mini-apps establish stable source conventions, a Template Registry may pair immutable cloneable source revisions with their authoring Skills, and a Mini Store may let a Human choose one. Instantiation then performs a complete bounded text-source copy into a new independent Mini-app draft and records origin provenance. Template inheritance, overlays, and automatic upstream merging remain out of scope.
- Dashboards and Mini-apps expose intentional named **Surfaces**. The initial contract has separate `preview` and `content` Surfaces that may have different compositions and information priorities. A Dashboard Surface owns a declarative Render Spec; a Mini-app Surface names a qualified local module entry in its source revision. A Surface is owned within a Dashboard or Mini-app revision, not a standalone entity with another lifecycle, permission model, or revision graph.
- A **Placement** pins a live reference to one Surface by `{ownerRevisionId, surfaceName}`. It stores destination, order, size, and local parameters; it does not copy the Surface or its data. Moving or removing a Placement does not change the source artifact's identity, lifecycle, provenance, or permissions.
- A Placement may store validated presentation parameters such as size, order, period, branch, or comparison mode. It cannot edit source code or a Data Query. Shared parameters are versioned with the shared Placement; Actor-scoped personal filters do not mutate the shared Dashboard. A structural composition change creates a new Dashboard draft revision.
- Declarative Dashboard Surfaces use only `@json-render/core` and `@json-render/react` as an internal implementation of the Render Spec and catalog/registry boundary. A deliberately small Autopilot catalog maps exclusively to public `@questpie/ui` components. Autopilot does not adopt `@json-render/shadcn`, json-render code generation, or a parallel client data/action runtime. json-render does not define the domain model, own durable data, execute arbitrary fetches, or bypass QUESTPIE authorization. Render data comes from authorized TanStack Query snapshots; generated specs may invoke only named QUESTPIE commands through qualified action handlers. A compiled Mini-app module project is not forced through this declarative catalog.
- The generic **Mini-app Host** supplies only isolation and product integration: null-origin `iframe sandbox="allow-scripts"`, host-controlled runtime dependencies and CSP, size limits, source validation, a short-lived Actor/session/Mini-app/revision/Surface-bound bridge token, preview/content mounting, governed Data Query access, and resolved Integration Action dispatch. The cloned Template code supplies client-side domain behavior such as slide navigation, presenter mode, lightbox, local editing affordances, and export UI. It never receives credentials or ambient QUESTPIE authority.
- Every Mini-app declares a qualified **UI Profile**. `product` is the default for internal tools and uses the versioned Mini-app UI Runtime derived from canonical public `@questpie/ui` components for application chrome and interactive controls. `canvas` is an explicit exception for Slides, wireframes, visual documents, and branded output: Host chrome and editing controls still use the canonical runtime, while the declared canvas/content region may use its own visual kit. A Mini-app cannot import Base UI, shadcn source, arbitrary npm UI packages, or a parallel home-grown product primitive system at runtime.
- The **Mini-app UI Runtime** is a small host-controlled, browser-compatible projection of `@questpie/ui`, not a second design system or an authorization boundary. The Host pins a compatible runtime revision, injects semantic design tokens and the resolved light/dark theme into the isolated frame, and exposes a bounded component/icon catalog. Product-profile qualification reports raw replacement controls and may block publication when an app recreates canonical buttons, fields, selections, menus, dialogs, status, loading, or error primitives. Preview Surfaces remain compact and overlay-free; complex overlays belong to the full content Surface so they are not clipped by an embedded frame.
- Every Mini-app revision contains one required, statically validated `app.json` **Source Manifest**. It declares the manifest schema version, named Surface entry modules, UI Profile and runtime compatibility, and symbolic Data Query and Integration Action requirements exported by Integration Packages. The owning Integration Installation selects the active package version through its explicit Dependency Policy. A `.app` project has no `server.ts` action registry: reusable server computation and every Company-data or external effect are first-class Integration Actions. Convention may provide defaults within the manifest contract, but convention alone never determines authority or the published runtime surface.
- Canonical text source lives in a first-class Mini-app draft/revision as a bounded logical file map; it is not stored as polymorphic Assets or in `document_store`. Binary inputs are Asset references. Publishing makes the source revision immutable and creates a content-addressed execution bundle for the host/sandbox. `document_store` remains capability-granted runtime records owned or shared by Mini-apps.
- A Mini-app may offer direct browser download for a bounded client-only export. Publishing an export into the Company creates an authorized Run over one exact Mini-app revision and writes the PDF, PPTX, HTML bundle, image, or other output as an Asset linked to that source revision and Run. Autopilot never needs to download a file and upload it again merely to cross this boundary.

### Data Source and generated read plane

- An **Integration Installation** is the authenticated Company binding to one Integration Package revision. It owns Credential Bindings, activation health, and granted capabilities. A **Data Source** is a named readable projection exposed by that Installation over a database, API, warehouse, lake, file store, spreadsheet, event source, MCP server, or QUESTPIE collections. An Action may use the same Installation without belonging to its Data Source.
- From the owner's perspective Data Source setup is a blank-page, outcome-first conversation. Autopilot may generate the required integration rather than requiring QUESTPIE to ship a hand-written connector first.
- An Installation exposes reusable, versioned **Integration Adapters** (formerly Source Adapters). An Integration Adapter is executable code that Autopilot or an authorized Human may author; qualification validates its host, import, credential, schema, cost, and capability contract before activation. One adapter owns one integrated system's complete transport contract in one definition: named credential requirements, health, schema discovery, typed read functions for the Data Sources it exposes, optional source-change ingress, and sealed effect functions. Authentication and transport logic are declared once per system, never per widget or per Action.
- Adapter effect functions are a sealed block that only Integration Action Runs may invoke. The Query execution paths never receive the effects capability, so read/write separation is enforced by execution authority and the Operation profile rather than by file placement or a separate effect-transport definition kind.
- A Data Source is statically declared by its Integration Adapter and identified by Installation, adapter Package Export, and declared source key (for example `shopify.orders`). There are no runtime source instances or parameterized source templates: schema discovery informs authorship, so a generated adapter declares business-level projections for the concrete system it was written for, and a Space subset is granted per declared Data Source. One Installation is exactly one authenticated account context — a second Shopify shop or a second database is a second Installation of the same package, while multiple named credentials inside one Installation are roles of the same account context, never different accounts.
- A **Data Query** is a separately versioned read-only Operation over one or more Data Sources, qualified Adapter functions, or other Data Queries. It records typed parameters, expected result schema, limits, provenance, validation evidence, and a dry-run result. Query code may compose reads and bounded transforms directly; V2 does not introduce a separate Data Pipeline entity. Reusable composition is another named Data Query, and only a Query that intentionally exposes a reusable source-like contract becomes a derived Data Source. Dashboard clients never join protected source results in the browser.
- Query and Action source receive generated typed `queries` and `source` proxies derived from the active Integration Package exports. Calling a Query proxy preserves the Query Runtime boundary; calling a Source proxy preserves the Adapter boundary. Pure local helpers use ordinary relative module imports and execute inside the caller's bundle. First-class Query or Adapter implementations are never imported and invoked as raw handlers merely to bypass authorization, limits, result policy, or provenance. This keeps generated code ordinary and composable without string operation names or hand-authored revision ids.
- Proxies are delivered exclusively through the handler context, never through imports or string keys: `ctx.source.<sourceKey>.<fn>` crosses the Adapter boundary, `ctx.queries.<packageExport>` crosses the Query Runtime boundary, and `ctx.effects.<name>` exists only inside Integration Action execution. Package codegen — the same engine family as QUESTPIE's `.generated/` registries — emits the context types from the declared tree on draft save, so generated code is fully typed without runtime module resolution, and the host resolves `ctx.*` calls to exact revisions per the Execution Snapshot. Mini-app source consumes a generated typed bindings module (`bindings.queries.<alias>`, `bindings.actions.<alias>`) whose calls the Mini-app Host bridge resolves to exact published revisions. Boundary calls are therefore visually distinct from ordinary `lib/` imports in both package and Mini-app code.
- The Query Broker selects the execution path by code provenance alone, recorded at qualification as `builtin_trusted` or `generated`; the shape of a Query never selects the path and there is no declarative fast-path authoring form. Generated package code always executes in the read-only Deno sandbox: one Query Execution is one guest evaluation of the query handler, `ctx.source.*` calls execute the adapter functions bundled into the same guest under the same Execution Snapshot, and each `ctx.queries.*` call is a governed host round-trip that becomes its own Query Execution with its own result policy, cache reuse, and Result Scope. The built-in first-party adapter over QUESTPIE collections is trusted host code and reads natively host-side under the same governed limits, scope, and audit. Neither path receives effect capabilities. A Surface Binding references a published Data Query revision; raw query text does not live in a widget and is not regenerated on every refresh.
- Autopilot may author source-specific queries when its own Agent authority grants schema inspection and query access. A browser widget never receives a credential or executes arbitrary SQL, URL fetches, or Integration Adapter code directly. A server-side Query Broker executes the published Data Query through the active Integration Adapter revision with RBAC, cost, row, size, timeout, and audit limits.
- Every Data Source declares a read availability strategy: `materialized`, `live`, or `hybrid`. Each Surface Binding declares refresh intent such as `realtime`, `invalidate`, `interval`, `manual`, or `snapshot`; it does not select a low-level execution mechanism. The server-side **Data Query Runtime / Refresh Coordinator** derives `request`, `schedule`, or `source_change` execution triggers from the Source strategy, refresh intent, result policy, and active grants. It owns due times, shared-result deduplication, retries, and Query Execution history, so two Surfaces bound to the same published Query do not create competing refresh loops. Product Automations and Workflows do not own Dashboard refresh; they are a separate business-work domain, although they may deliberately consume the same data events later. TanStack Query owns client snapshots and reconciliation. Components only render snapshot, freshness, loading, stale, degraded, and error state.
- Realtime external change enters through an Integration Adapter webhook, CDC, stream, or polling ingress. The Data Source Runtime normalizes it, resolves affected Data Queries, invalidates or refreshes their results, persists any required result revision, and then emits a bounded QUESTPIE Channel result-change event. A Source never pushes unfiltered external rows directly into a browser Surface.
- The adapter's `ingress` block supplies handlers only: webhook `verify` and `parse` run in the guest behind a host-owned per-Installation ingress URL, and `poll.run(cursor)` is invoked on a Refresh Coordinator-owned cadence with runtime-persisted cursors, active only while an Active Materialization exists. A webhook signing secret is an ordinary named Credential Requirement. Normalized source-change events are coarse per-source invalidation hints (`{source, hint?}`) rather than row-level streams; the coordinator resolves dependent Queries through the qualification-time dependency graph, and row-level granularity is deferred. The built-in package's ingress is a native mapping of QUESTPIE collection change events, so the first Goals/Tasks slice is live without webhooks.
- Every Query Binding also declares a result policy: `ephemeral`, `cached`, `materialized`, or `historical`, with explicit retention, encryption, lineage, and audience rules. Persisted results record the Integration Adapter revision, Data Query revision, acquisition time, source freshness, and authorizing Data Projection Grant. External query results are neither retained forever by default nor forced to refetch for every view.
- Two Bindings may reuse one Query Result only when they have the same published Data Query revision, normalized parameter values, result policy, Source and adapter revisions, and **Result Scope**: Company/Space scope, Projection Grant or Actor authority, audience, field mask, data classification, and row-policy identity. A personal read is Actor-scoped unless an explicit shared Data Projection Grant produces the same Result Scope. A Result produced under broader authority never becomes a cache entry for a narrower or different audience merely because its Query and parameters match.
- When a refresh fails, the Runtime may serve the last successful authorized Result as `stale` or `degraded` with its acquisition time, source freshness, bounded failure state, and retry status. It must revalidate the current Result Scope before every delivery. Revoked or expired authority, broader data classification, changed field or row policy, an incompatible schema, or changed metric semantics blocks the prior Result instead of presenting it as stale. A transient Source or execution failure does not erase a still-authorized last successful Result.
- Every published Data Query revision defines a safe upper `maxStaleness` bound as part of its Freshness Contract. A Surface Binding or Data Projection Grant may require the same or a shorter bound, including zero tolerance, but cannot widen it. Once the effective bound expires, the Runtime returns an unavailable result state rather than the prior value; retention of a historical Result does not make it eligible for current presentation. The Surface exposes the result's `asOf` and freshness state whenever business interpretation depends on them.
- Surface Bindings refresh independently by default. A Dashboard revision may place semantically coupled Bindings into a named **Consistency Group** with a common `asOf` contract. The Runtime publishes a new group revision only when every required member satisfies that contract; it never presents a partial mixture as a coherent group. If one member fails, the last complete authorized group revision may remain degraded within the strictest effective Freshness Contract, after which the group becomes unavailable. Unrelated Bindings do not inherit this coordination cost.
- Background and `source_change` refresh is maintained only for an **Active Materialization**. The initial rule deliberately stays simple: a valid published Placement or other published shared Surface dependency makes its exact Result Scope active; personal/ad-hoc and unused Results are only invalidated and recompute on demand. Removing the last published dependency stops proactive refresh immediately, while any retained Result remains subject to its existing retention and Freshness Contracts. Active Materialization is derived coordinator state in the first implementation, not another author-managed object or a separate scheduler product. The coordinator still coalesces duplicate source changes for the same scoped result.
- Schema inspection, sampling, and querying are separate grants. Schema metadata may be disclosed without row data. Sampling requires an explicit grant, field/row limits, and redaction before data reaches model context. Qualification may execute a query without disclosing raw results to the Agent.

### Effects, MCP, and Skills

- **Operation** is the internal common contract family for versioned executable definitions. Data Query and Integration Action are sealed profiles sharing identity, typed input/output, qualification, provenance, and binding mechanics. Query is provably read-only and owns Query Result semantics; Action is effectful and owns approval, idempotency, and Run semantics. They are never distinguished by a mutable `mutates` flag.
- Data Sources and Data Queries form the read plane. They never hide a write operation inside a query.
- An **Integration Action** is the canonical write/effect definition. It has versioned typed input/output, effect classification, idempotency semantics, required grants, approval policy, and audit behavior. It is independent of any one Data Source and may use zero or more Integration Installations, Data Sources, and Data Queries.
- An Action declares typed input and output schemas, an effect classification from the closed set (`company_data`, `external_write`, `destructive`, `financial`, `access_control`, `cross_space_disclosure`), and an optional idempotency derivation over its validated input; without one, the Run id is the idempotency key. `company_data` is the baseline class for ordinary governed Company-data writes — plain RBAC applies and no Review Gate is required — which preserves the rule that Dashboards and Mini-apps mutate Company data only through a named authorized Action. The remaining classes align with the Review Gate effect families. Delivery is at-least-once: the host checks the key before dispatch, a duplicate delivery or retry with the same key returns the recorded result instead of re-executing the effect, and adapter effect functions receive the key to forward into external idempotency mechanisms where the target system supports them. For systems without an idempotency API this guard is best-effort, and the declared classification states whether replay is safe. Validated output is recorded on the Run; artifacts are Assets linked to the Run and revision.
- A Mini-app invokes an Integration Action through a qualified **Integration Action Binding** whose local alias resolves to one exact published Action revision. An Agent invokes the same Integration Action through its MCP tool adapter. These are two presentation/invocation paths over one command contract, Run model, authorization decision, approval policy, and audit trail, not separate endpoints with divergent behavior. Pure client-side UI handlers remain ordinary imported Mini-app code and are not Integration Actions.
- Generated Action implementations always execute through the sandboxed Deno executor. Their source declares symbolic Query and capability dependencies; qualification resolves them into typed bindings. The trusted QUESTPIE host validates the exact Action revision and input, checks RBAC and effect approval, resolves credentials and capability grants, creates and links the Run, and dispatches execution. An Action is not a QUESTPIE Job or Workflow definition: a generic durable job/worker may deliver a Run to Deno, while Workflow remains an optional future orchestrator. The current Dashboard and Mini-app scope exposes only an explicit immediate invocation through a qualified Binding; delayed, child, recurring, and event-driven delivery are not part of this implementation slice.
- An integration **Skill** is the Agent-facing playbook describing when and how to combine Data Queries and Integration Actions. A Skill does not replace Agent RBAC, Source grants, effect policy, or review gates.

### Deferred Automation compatibility

- Automation, Flow authoring, cron delivery, delayed invocation, and child Action dispatch are outside this ADR's current Dashboard, Mini-app, Data Source, Data Query, and Integration Action implementation scope. They must still remain compatible with the first-class versioned Action boundary; no Mini-app-local server or scheduler API may preclude a later governed invocation adapter.
- In the wider Autopilot domain, an **Automation is not a cron-to-Action shorthand**. It combines a durable Trigger with a repeatable Work Template. Each firing snapshots a new Work Request whose target may be adaptive Agent Work, a versioned Flow, or a direct Action. For example, “Every Friday at 14:00, perform a full competitor research and publish the result to Company Knowledge” creates work for an authorized Agent, which may plan and use Skills, Data Queries, and Integration Actions during its Run.
- An Automation never grants the target Actor ambient authority. Every read and effect remains subject to the Actor's current grants, Action policy, approval, and audit boundary. The exact Automation/Flow/Work Request model will be decided in its own later phase rather than smuggled into `app.json`, a Dashboard refresh policy, an Action implementation, or a queue job name.

### Packaging, installation, authority, and revisions

- A versioned **Integration Package** groups Integration Adapters, Data Queries, Integration Actions, MCP exposure, Skills, and optional Mini-app Templates that were designed and qualified together.
- The canonical Integration Package source tree mirrors QUESTPIE's by-type file conventions. First-class published definitions live one per file as default-exported definitions in `adapters/`, `queries/`, `actions/`, `skills/`, and later `templates/`; the kebab-case filename derives the stable camelCase **Package Export** name that bindings, proxies, and Skills reference. Everything else — `lib/` modules, `_`-prefixed entries, colocated `*.test.ts` files — is ordinary private code imported relatively and bundled into its caller; it never becomes a published entry. Typed input/output schemas are declared inline in each definition; shared shapes are ordinary `lib/` imports.
- There is no in-tree package manifest. Package identity, version, and description live on the Integration Package record, and publication derives the export map from the tree by convention; a manifest would only duplicate the tree and introduce drift. Data Sources are declared by Integration Adapters rather than as standalone files. The same logical tree convention applies to the repo-hosted built-in package and to generated packages stored as draft/revision file maps.
- A Company owns an **Integration Installation** that binds Company-specific credentials, capabilities, health, Runs, audit state, and one explicit **Dependency Policy**. `automatic` always selects the latest published Package version; `pinned` selects one exact Package version. There is no inferred compatibility class or per-consumer dependency lock. Package contents remain individually grantable, and changing versions never grants a new capability.
- Personal or ad-hoc Dashboard reads execute under the current Actor's authority. A shared published Dashboard requires a bounded **Data Projection Grant** naming the exact Data Query revision, fields, Space or Company scope, audience, refresh policy, and expiry/revocation state. A Dashboard never borrows the ambient authority of its creator or generating Agent.
- An Agent may create and execute private/ad-hoc Data Queries within its own Source grants and execution budgets without per-query Human approval. Publishing a Query or Projection for a shared audience requires either a specific Human approval or an explicit pre-existing Agent grant such as bounded query/projection publication authority. An authorized Agent publishes under its own identity and leaves full provenance; delegated autonomy satisfies the gate rather than bypassing it. Without that authority the operation creates a Permission Request.
- Agent publication autonomy is bounded by Integration Installation or Data Source, allowed Spaces and audiences, data classifications, query cost and refresh limits, and optional expiry. It is not a blanket Company-wide bypass. A policy may, for example, let Autopilot independently publish internal read-only inventory Dashboards in one Space while excluding personal data, new external hosts, cross-Space projection, and all effects.
- Generated Integration Adapter and Integration Action code is immutable and revisioned. An update creates a draft revision; qualification, capability diff, and promotion occur before it becomes active. The prior revision remains active until promotion, and rollback is a forward publication of a previously qualified version's content.
- Package versions are monotonic integers with an optional human label; there is no semver and no declared compatibility semantics. Publication is an atomic immutable snapshot of the whole package tree in which every Package Export receives a content-addressed revision, so an unchanged file keeps its revision identity across versions. Rollback publishes a new version with a prior version's content — the latest pointer never moves backwards and history stays linear. The default Dependency Policy for company-owned generated packages and the built-in package is `automatic`; `pinned` is a deliberate choice, and a default for future third-party registry packages will be decided when they exist.
- Authors and generated source refer to stable symbolic exports rather than hand-editing revision ids. An automatic Installation advances atomically whenever a new Package version is published; a pinned Installation does not. Publication qualification belongs to the Package release process and never guesses whether one Installation should update. Every Query Execution and Action Run records an immutable **Execution Snapshot** containing the exact Package, Operation, Adapter, and bundle revisions it used, so in-flight work and provenance never move with the Installation's latest pointer.
- New hosts, credentials, data fields, write capabilities, or broader scopes never become granted merely because an automatic Installation advanced. The affected Operation remains unavailable or requests explicit permission under the ordinary authorization model.
- Source schema drift never mutates a published Data Query revision. Autopilot or a Human prepares and explicitly publishes a new Package version. Automatic Installations then follow it and pinned Installations do not; consumers that cannot satisfy its contract become visibly degraded rather than triggering a hidden compatibility heuristic.

### Initial proving slice

- The first end-to-end proof uses Autopilot's own QUESTPIE collections as a built-in Integration Installation and Data Source. A grounded Goals/Tasks Query feeds one small Dashboard; one explicit Action and one Mini-app consume the same qualified bindings. This slice must prove RBAC, Query refresh and Channels reconciliation, typed bindings, immutable publication, Action Run state, and canonical UI before an external credentialed Adapter is attempted.
- Concretely, the built-in package `autopilot` (provenance `builtin_trusted`) ships `adapters/autopilot.ts` exposing `goals` and `tasks` Data Sources with native collection-event ingress, no Credential Requirements, and a `completeTask` effect; `queries/goal-progress.ts` returning criteria progress and task status counts for one Goal; `actions/complete-task.ts` classified `company_data` with input-derived idempotency; and `skills/autopilot-data/SKILL.md`. One Dashboard ("Prehľad cieľa", one content Surface) is placed into a Space, and one Mini-app (`goal-checkin.app`) consumes the same `goalProgress` and `completeTask` bindings, proving UI and MCP parity over one contract. The slice deliberately instantiates every architectural concept from this ADR exactly once.

### Credentials, storage, Drive, and sandbox execution

- A generated adapter declares named **Credential Requirements** from a closed framework-owned kind set (`api_key`, `oauth2`, `basic`, `dsn`, `service_account`), each with constraints, optionality, and a plain-language purpose. An adapter with no requirements — such as the built-in package — still activates as an Installation, which remains the activation and grant anchor. Credential Binding is one concept shared with AI Provider Connections, not a parallel integration-specific secret object. Only a Human with Installation management authority may satisfy a requirement with a revocable Credential Binding; an Agent's request creates a Permission Request, and raw secret material is never exposed to Autopilot, prompts, source code, Run records, logs, or browser state. Rotation changes the binding revision and rechecks health; it does not rewrite source.
- Secret material reaches executing adapter code only through narrow injection into the Deno guest at execution time, accessed exclusively through the `ctx.credentials.<name>` accessor rather than raw globals. The host resolves the Credential Binding per execution and injects the narrowest usable material per kind: `oauth2` injects only a short-lived access token while refresh tokens and OAuth client secrets never leave the host; `api_key` and `basic` inject the bound value; `dsn` and `service_account` inject connection material consumed by an in-guest driver. The guest's default-deny egress allowlist is limited to the adapter's declared hosts, so leaked material could at most travel to the system it belongs to. The accessor seam deliberately permits a later per-kind host-side authenticated broker without rewriting adapter source.
- The trusted host owns the complete OAuth lifecycle: authorization callback, code exchange, token storage, and refresh. An `oauth2` Credential Requirement only declares provider endpoints, scopes, and purpose. For generated integrations the Company registers the OAuth application and a Human supplies its client id and secret during binding setup; platform-registered applications may later serve curated packages.
- Credential Bindings are resolved at execution time, never at enqueue time, so rotation is harmless to queued work and each Execution Snapshot records the binding revision actually used. Rotation is not revocation: a cached Query Result produced under then-valid authority continues to be served under the existing Result Scope revalidation and freshness rules, while revoked or expired authority blocks it. Credential expiry or revocation moves the Installation into a visible credential-expired health state: dependent Queries serve Degraded Results under the existing contract, and a new or queued Integration Action Run fails fast with a visible `credentials_expired` state and a rebind call-to-action. Retry after rebinding is a new linked Run; Action Runs do not pause and auto-resume awaiting credentials.
- Canonical small source and declarative spec revisions live in versioned database records for diff, provenance, and audit. Activation produces an immutable content-addressed execution bundle stored as a private upload Asset. Sandbox execution selects the exact qualified bundle hash, not mutable draft text.
- The QUESTPIE upload-enabled `assets` collection remains the physical file/blob metadata boundary. It is not a polymorphic replacement for Knowledge, Skills, queries, integration revisions, Run artifacts, and Mini-app runtime records.
- `document_store` remains namespaced runtime record storage for Mini-apps. It does not store executable source, package identity, Knowledge, or Drive navigation.
- The Company Drive contains typed references/projections to canonical Knowledge, Assets, Data Sources, Data Queries, Integration Packages, Skills, Dashboards, Mini-apps, and related revisions. It does not copy those entities, and a Drive move does not change runtime identity or permissions.
- Untrusted generated executable code runs only through QUESTPIE's sandboxed executor under a default-deny manifest: exact network/import hosts, file paths, data operations, services/jobs/workflows, timeout, memory, credential bindings, and short-lived authority. Declarative Render Specs and Skills are not treated as arbitrary executable scripts.

### Non-technical setup contract

- Setup is conversational and outcome-first. The primary path asks what the Company wants to connect and achieve, requests only unavoidable credential binding or business-meaning input, shows a grounded data preview, and explains requested access and effects in plain language.
- Initial activation of a generated read-only integration always ends in one explicit Human commit over the plain-language data contract and grounded preview. Passing sandbox qualification alone does not activate it. Every later Package version is explicitly published; automatic Installations follow the newest published version and pinned Installations do not. Publication and Company authorization remain separate decisions.
- Code, host allowlists, schema details, query plans, capability manifests, execution budgets, and revision diffs remain inspectable under progressive technical disclosure. They do not become mandatory concepts for the target operator.
- Safety is never hidden: the operator sees which system will be read, which business data will be shared with whom, which external actions may occur, whether approval is required, freshness, and how to disable or revoke the integration.

## Framework basis and rejected legacy shape

QUESTPIE already provides upload-enabled collections, local/S3/R2-compatible storage, signed private URLs, `f.upload()` relations, sandboxed Deno execution, and default-deny capabilities for network, imports, files, collections, namespaced stores, services, jobs, and workflows. These primitives should be extended rather than replaced.

The previous Autopilot branch collapsed uploads, Knowledge text, Skills, Mini-app source, Run logs, results, and previews into one polymorphic `assets` collection, while its `scripts` collection stored mutable runner/content/enabled fields without revision, provenance, capabilities, qualification, or promotion. V2 explicitly rejects both shapes. Physical blobs, immutable artifact revisions, operational domain entities, Drive projections, and Mini-app runtime records remain separate.

## Consequences

- Autopilot can generate integrations for systems QUESTPIE has never seen while the runtime remains bounded by a small host contract and explicit capabilities.
- Dashboard previews, full dashboards, and Mini-app previews can share governed data without sharing lifecycle or effect authority.
- Shared dashboards have predictable authorized projections instead of silently inheriting creator access.
- Generated integrations require qualification infrastructure, capability-diff UX, Query Broker limits, versioned execution bundles, credential binding, and clear degraded/freshness states.
- The Phase-2 implementation must define collection schemas, driver contracts, publication commands, migration and revocation semantics, negative security oracles, and end-to-end operator scenarios before production code begins.
