# Builder vibecoding and managed AI compute

Autopilot Phase 3/4 ships a **Builder**: Lovable-class vibecoding over a Project's real Git repository, with live Preview, human approval, and audit, plus config-driven Deploy. It must work EZ in two substrates with one contract — a self-hosted dev machine (zero cloud dependencies) and managed production (single-deployment row-level multi-tenancy, shared Worker Pools, ephemeral per-Run sandboxes). A third, product-critical path is a **non-technical customer connecting their own computer** as a `manual` Work Machine, so their work runs locally and incurs no managed-compute charge (connected-PC-first scheduling); enrolling that machine must require near-zero manual work on it — no Docker, no terminal, no hand-edited config. This ADR records the ratified decisions from the Builder + managed-AI-compute architecture grill (grounded by a four-lens research run with dual adversarial review). Every agenda question is resolved and the owner has **accepted** it; implementation is a later phase (Phase 3/4).

The load-bearing principle is a **uniform-environment invariant**: it must not matter *where* the harness runs — customer PC, self-hosted server, or managed pool — the harness always runs against the same sandbox contract. One environment shape, N substrates.

## Decision

### Builder execution primitive

The Builder session's execution primitive is the Vercel AI SDK harness **`HarnessV1SandboxProvider`** seam (`@ai-sdk/harness` + `Experimental_Sandbox{Session,Process}` from `@ai-sdk/provider-utils`) — the same interface that `questpie-cms/packages/ai`'s `createLocalHostSandbox()` already implements for localhost. It is the only execution contract that carries the four things a Builder conversation needs across turns: multi-turn shell state, arbitrary file I/O, a long-running dev server, and exposed preview ports. A managed sandbox is one more class satisfying this interface, not a harness redesign.

This is explicitly **not** the `@questpie/sandbox` Deno executor that ADR 0017 mandates for generated Mini-app and Integration code. That executor is single-shot request/response with no persistent session, no filesystem API, and no port exposure — structurally incompatible with a Builder dev loop. The two execution planes are deliberately separate and this separation is recorded in writing precisely because ADR 0017 (proposed one day earlier) makes the Deno executor look like *the* pattern for running generated code: ADR 0017's executor runs bounded, read-only or effectful generated Operations; the Builder runs a stateful interactive coding Harness. A new Builder-owned execution abstraction is rejected as a KISS / upstream-first violation — the harness seam is already shipped upstream.

### Localhost and self-hosted Default Work Machine

The localhost and self-hosted Default Work Machine dev loop is backed by the existing `createLocalHostSandbox` implementation — a `node:child_process` `bash -lc` executor with per-session HOME isolation, an env allowlist, secret scrubbing, and a single loopback preview port. It requires no Docker or container runtime on the dev machine (only PostgreSQL, via an optional compose file). This is the "EZ localhost" story: the Builder works with zero cloud dependencies.

Parity with managed production is deliberately located at the `HarnessV1SandboxProvider` interface, not at the isolation mechanism. Requiring Docker locally for "parity" is rejected because it breaks EZ-localhost for no gain; a browser-side runtime (WebContainers) is rejected as closed-source/commercial for for-profit production and a divergent execution model that would break substrate uniformity.

`createLocalHostSandbox` is a trusted single-user executor: it provides no VM/container isolation, so it must never serve as a multi-user fallback. ADR 0001's worker trust boundary already enforces this — hosted/multi-user admission fails closed and must never fall back to the local-host adapter.

### Managed-cloud sandbox provider

For `questpie_managed` cloud, `HarnessV1SandboxProvider` is implemented by a **self-hostable OSS microVM/container runtime wrapped behind the upstream interface** — not by `@ai-sdk/sandbox-vercel`. The upstream-first win is the *interface*: `@ai-sdk/harness-claude-code`'s own README names `@ai-sdk/sandbox-vercel` as "the supported choice today," but Vercel Sandbox has no self-hosted equivalent (Vercel-project OIDC only), which would break the uniform-environment invariant — self-hosted (`manual` / `guided_provider`) and `questpie_managed` must run the *same* provider. ADR 0013 already rejected both "one permanent container per customer" and a cloud-only fleet, so the managed implementation cannot be vendor-locked. Building a pooled provider from scratch on QUESTPIE's own K3s is rejected as a KISS violation — wrap a proven OSS runtime, don't build a sandbox platform.

The specific OSS runtime is deliberately left open (open-preference), chosen on the ops-appetite-vs-isolation-strength axis when the pooled provider is actually built. Research candidates: E2B-infra (Apache-2.0, Firecracker; heaviest/most-proven; Terraform self-host), microsandbox (Apache-2.0, libkrun; lightest, sub-200ms; least-proven), and agent-sandbox (OSS, K8s, E2B-wire-compatible; aligns with ADR 0013's K3s Worker Pool). Daytona is rejected as a base (went closed-source June 2026).

Sequencing is explicit and load-bearing: ship **localhost + single-machine self-hosted first** (the `createLocalHostSandbox` path above), and treat the pooled remote provider as **deliberately deferred**, gated on the worker enrollment/identity hardening decision below. Two framework gaps must close before the pooled provider ships: the sandbox provider must become *pluggable* (`harness-core.ts` currently hard-codes `createLocalHostSandbox` with no substitution seam), and the pooled provider must add cross-process port/session coordination (the harness's `bridgePorts` lease is a process-local WeakMap by design). A remote provider must also return a genuinely remote-reachable `getPortUrl` and implement `setNetworkPolicy` (reusing `@questpie/sandbox`'s SSRF egress allowlist) and `setPorts`.

### Worker enrollment, identity, and machine connection

Worker enrollment and workload-identity hardening is a **hard gate** before any multi-worker topology ships — the managed pool and a self-hosted operator's second `manual` Work Machine alike. The current holes block both: enrollment/register routes carry no access rule (effectively public), worker identity is not bound to the authenticating secret (poll/heartbeat/deregister trust a client-supplied `workerId`, so any valid secret can act as or deregister a different worker), `ai_workers` carries no owner/Company/attestation, and there is no finalize-over-HTTP path for a remote worker to report terminal Run state. Because these break a self-hosted second machine too, "self-hosted" is not automatically the safe fallback.

The fix lands **upstream** in `@questpie/ai`'s `WorkerManager` — the one fleet implementation to converge on — adding route access rules, secret-bound worker identity, owner/Company/attestation on `ai_workers`, and finalize-over-HTTP. The duplicate `questpie-agent-runtime` (design docs only, zero commits) is retired as an input, and `apps/autopilot`'s `ai-worker.ts` is superseded by the upstream manager. This is a prerequisite for the pooled provider and for Draft scheduling, not parallel work.

The same enrollment flow is also the **zero-friction "connect your PC" path** for a non-technical customer using a `manual` Work Machine (connected-PC-first): a downloadable agent + a one-time enrollment token + a browser sign-in, requiring no terminal, Docker, or hand-edited config on the customer's machine. Security and low friction are not in tension — the one-time token carries the binding that hardens identity, and the installer performs the Runtime Adapter setup (e.g. the coding CLI) the customer would otherwise do by hand. The exact installer packaging (native installer, background/tray app, or a single paste-command) is open-preference; the binding constraint is near-zero manual work on the enrolled machine.

### Draft workspace affinity

A Builder Draft's live workspace is **Work-Machine-affine**: it pins to one Worker Home, and scheduling protects the stateful session rather than cloning it. ADR 0013 already requires this — a stateful Harness Run is pinned to the compatible Worker Home and may move only when its Runtime Adapter declares and successfully performs a supported checkpoint/resume transfer — and no adapter performs that transfer today (`createLocalHostSandbox`'s `resumeSession` re-derives the per-`sessionId` directory on the same disk; live attach is impossible by design). `apps/autopilot`'s worktree-per-Draft design is likewise single-disk by construction.

Free pool-scheduling is rejected: it silently assumes floating Drafts that ADR 0013 forbids and needs cross-worker file sync and a session-transfer contract that does not exist. Stateless re-clone per turn is rejected: it loses the shell state and in-progress changes a vibecoding session depends on. When faster cold-start is genuinely needed, the harness's existing identity/snapshot machinery (`prepareHarnessSandboxTemplate`, `hashHarnessBootstrap`, `applyBootstrapRecipe` — currently unused; `createLocalHostSandbox` ignores the `identity` param) is wired, rather than inventing a parallel snapshot cache. Affinity granularity — per-Draft vs per-Project Worker Home — is open-preference.

### Preview URL provider

A Draft's internal dev-server port becomes a browser-openable **Preview URL** through a new upstream-owned preview-URL provider behind a stable interface — not by piggybacking on whichever sandbox vendor the pooled provider wraps. This is the single emptiest seam today: nothing anywhere resolves an ephemeral sandbox's internal port to a public HTTPS URL. The provider maps `{session, internal port}` to a public, per-session, TLS-terminated URL and routes both HTTP and the Vite HMR WebSocket (it must honor `server.hmr.clientPort` and `allowedHosts`).

Two substrate implementations sit behind one interface, preserving the uniform-environment invariant: localhost uses the existing loopback `getPortUrl` (`http://127.0.0.1:<port>`), and managed / self-hosted-server uses an ephemeral per-session wildcard-subdomain reverse proxy with TLS, reusing `questpie-cloud`'s proven Traefik + wildcard-DNS + TLS pattern (today proven only for long-lived per-deployment Services) adapted to per-session routing. Binding preview to the sandbox vendor's own URL scheme is rejected because it re-couples preview to the vendor the previous decision deliberately avoided and breaks parity when managed and self-hosted use different backends. The v0 static-artifact-snapshot model is rejected because it cannot serve a live vibecoding dev server (v0 explicitly rejected live preview for a pull-based fleet behind NAT). The subdomain scheme and where preview-access authorization lives are open-preference.

### Repository layout and ownership

Each Project binds to an external **Project Repository** on the customer's git host (GitHub, GitLab, or generic Git), always reached as an HTTPS-only remote in every topology — which is what keeps localhost and managed at parity. On disk, the layout follows `apps/autopilot`'s hardened, path-clamped successor to the v0 workspace manager: one bare mirror clone per Project (`repos/<sha>.git`) and one git worktree per Draft, both inside the Draft's Work-Machine-affine Worker Home. The remote carries no embedded credentials, git is device/inode-pinned, and every workspace operation proves ownership with a marker.

Lovable-style GitHub-App two-way auto-sync is rejected as the on-disk/ownership model because it lacks the approval gate the mission requires (change delivery is decided separately below). A no-persistent-mirror PR/branch-per-session model (v0 / new-v0) is rejected because it forfeits the affine Worker Home and pays re-clone latency every session.

### Change delivery and the merge gate

A Draft's changes reach the Project Repository by **auto-push to the Draft's own branch**, with the human approval and audit gate located at the **Draft-to-main-branch merge**, not at every change. This is a deliberate, owner-chosen deviation from a per-change review gate: the product is called Autopilot and targets non-technical operators, so within a Draft the agent commits and pushes freely (frictionless, durable, and visible on the Project Repository), and multiple Drafts proceed concurrently as independent branches. The consequential boundary — merging a Draft into the main line — is the gated one.

That merge is the **Change Request**: a reviewable, auditable proposal that a Human approves in the Autopilot UI (diff + approve), never requiring the non-technical operator to visit the git host. `apps/autopilot`'s `change_requests` collection (providerKind `github`/`gitlab`/`generic_git`, `idempotencyKey`, status `creating`/`open`/`closed`/`failed`, one active Change Request per Draft) backs this merge boundary unchanged. Branch protection on the main branch enforces the same gate server-side, so approval cannot be bypassed even by a direct push; the Change Request may still open a native PR/MR under the hood for git-host-level auditability. Sending a Change Request or merging remains a side-effectful, permissioned action.

This moves the mission's "approval + audit" requirement from "every change delivery" to "merge into the main line" rather than removing it — the industry-converged GitHub-flow shape (feature-branch auto-push + protected main). Fully automatic merge-to-main with only post-hoc revert is rejected: it would let unapproved code reach the main line, removing the checkpoint the owner explicitly wants at the merge boundary. Native git-host PR review as the operator's approval surface is rejected for the non-technical path because it forces the operator into the git-host UI.

### Builder workspace domain shape

The Builder workspace domain is rebuilt in the **app** as product-meaning collections — `project_workspace_sessions` (the Draft, with its status machine and single-writer lease), `project_workspace_runs`, `project_workspace_checkpoints`, `preview_sessions`, and `change_requests` — following `apps/autopilot`'s recent, hardened, ADR-0013-adjacent schema. The sandbox, worker/fleet, and preview-URL primitives land upstream in `questpie-cms` and are consumed by the app. This is exactly the split the capability-reuse binding gate requires: the app owns product meaning, the framework owns reusable infra/execution.

The v0 shape — a polymorphic `assets` collection and a mutable `scripts` collection without revision, provenance, capabilities, qualification, or promotion — is rejected; ADR 0017 already rejects it explicitly. Pushing the whole Builder domain upstream as a generic framework module is rejected as over-generalizing product concepts (Draft, Checkpoint) into the framework, which the gate forbids. These collections stay behind system-mode `builderInternalAccess` — driven by the Builder's own commands and Runs, never direct authenticated-user CRUD. Whether checkpoint-artifact storage reuses the upstream `assets` collection or a domain revision table is open-preference (ADR 0017 leans revisioned, not polymorphic).

### Build and deploy

Deploy is modeled on ADR 0017's Integration Adapter / Integration Action machinery rather than as a hardcoded bridge to one cloud. A **Deploy Target** is a configured deployment destination backed by a deploy adapter — an Integration Adapter specialized for shipping a Project, owning that target's credentials and deploy effects. QUESTPIE Cloud is one Deploy Target (the managed default); Vercel, Fly, a self-hosted host, or a Company-authored custom target are others. Because deploy adapters follow the blank-page generated-integration model, a Company can author or customize its own deploy target exactly as it authors other integrations — this is the owner-chosen shape, and it keeps deploy on the same effect / adapter / Run / audit machinery as everything else instead of a bespoke deploy subsystem.

A multi-step deploy pipeline (for a complex target like Cloud: build → provision → apply → observe → verify) runs as a durable `@questpie/workflows` step graph that invokes the selected adapter's deploy effects; a simple target (e.g. Vercel) may be a single Integration Action. `@questpie/workflows` is shipped, already a v2 dependency, and questpie-cloud's own named orchestration target, so the pipeline reuses it rather than hand-rolled poll loops or an app-local state machine.

Build execution is therefore a **per-adapter** concern (Cloud uses its own buildkit/buildpacks; Vercel uses Vercel's build), which dissolves the question of a single global build executor. The questpie-cloud Deploy Target still honors Cloud's rule that Autopilot deploys through Cloud APIs, not by applying Kubernetes directly — the "don't apply K8s directly" constraint now scopes to the Cloud adapter specifically, while every other adapter uses its own target's API. `questpie-build-service` (buildah-based, ~10 months stale, unwired, needs a privileged pod, adoption undecided in Cloud) is not inherited infrastructure; it remains a deferred, optional build backend for a possible self-hosted / generic Deploy Target. Autopilot building its own K8s-applying deploy path is rejected as duplicating the questpie-cloud sibling and violating the adapter model.

The remaining unbuilt work is the concrete adapters and the Autopilot↔Cloud bridge (Cloud side: finish `/api/cloud/*` and a non-stub build executor; app side: a Cloud client, Project↔cloud mapping, a `cloud_deploy` action, an API key) — "the largest unbuilt strategic piece" per Cloud's own docs — now scoped as building the first (Cloud) Deploy Target adapter, not a parallel deploy system.

### Tenancy, quotas, and managed-compute pricing

The "single-deployment row-level multi-tenancy" premise is ratified here and its layer named explicitly, because it lived only in board knowledge before (no ADR carried it; `CONTEXT.md` even lists "tenant" as a word to avoid). Row-level multi-tenancy describes **Autopilot's own Company/Project/Run domain data**: one deployment and one database serve all Companies, the Company is the scope boundary, every collection read is scope-filtered, and cross-company mutation is blocked in the domain layer. This is a different layer from questpie-cloud's dedicated-DB/bucket/namespace per customer app — Cloud isolates the customer *apps it hosts*, not Autopilot's own store — so the apparent contradiction between the two is a category difference, not a conflict. Adopting per-tenant isolation for Autopilot itself is rejected: it discards the row-level cost model (base cost ~ rows, not tenant count) that Cloud's own vision doc endorses.

Managed **compute** is ephemeral per-Run on shared Worker Pools with per-Run authority and encrypted Worker Home — never standing per-tenant infrastructure. Isolation sits at the execution boundary (where the danger is), not the tenant boundary.

Managed-compute pricing is a **credits** model. One credit ledger meters managed compute (sandbox-minutes), storage, and platform-provided model tokens. Model tokens billed through a Company's own Provider Connection (its own API key) go directly to the Company and are not credits — only platform-provided models are metered in credits, preserving ADR 0013's rule that model API usage is always a separate ledger dimension from compute. A `manual` Work Machine (the customer's own connected PC) incurs no managed-compute credits (connected-PC-first). Margin is set in the credit price, and per-Company budgets, quotas, and stop policies apply (ADR 0013). Raw API-cost pass-through is rejected as a thinner value story with less predictable customer cost; flat subscription tiers are rejected as a poorer map of actual compute/model consumption. Concrete per-Company caps (max concurrent Drafts/Runs, session timeout, preview count, spend ceiling) and the credit-to-resource conversion are open-preference numbers, set as Work-Machine-capacity-aware defaults.

## Upstream framework work

Per the capability-reuse binding gate, these land in `questpie-cms` and are consumed by the app. Ordered roughly by dependency:

1. **Pluggable + pooled `HarnessV1SandboxProvider`.** Make the provider pluggable (`harness-core.ts` hard-codes `createLocalHostSandbox`), then add a second implementation over a self-hostable OSS microVM/container runtime with a remote-reachable `getPortUrl`, `setNetworkPolicy` (reuse `@questpie/sandbox`'s SSRF egress allowlist), `setPorts`, and cross-process port/session coordination.
2. **Preview-URL provider.** Map `{session, port}` to a public, per-session, TLS-terminated URL routing HTTP + HMR WebSocket, with a loopback implementation (localhost) and a wildcard-subdomain reverse-proxy implementation (managed / self-hosted-server) reusing questpie-cloud's Traefik + wildcard-DNS pattern.
3. **Worker enrollment + workload-identity hardening.** Access rules on enroll/register, secret-bound worker identity, owner/Company/attestation on `ai_workers`, finalize-over-HTTP; converge on `@questpie/ai` `WorkerManager`; retire `questpie-agent-runtime`. Gates all multi-worker topology.
4. **Durable AI execution service.** 3.16 ships worker/Harness infrastructure only; the generic durable `@questpie/ai` service with package-owned Run / Attempt / Command / Effect / recovery persistence does not yet exist. Builder Drafts and Checkpoints depend on it.
5. **Session checkpoint/resume transfer** (to relax Draft affinity later), plus wiring the harness's existing identity/snapshot machinery instead of a parallel cache.
6. **SecretStore + Provider registry/routing** for git-remote credentials, provider keys, and short-lived per-Run broker tokens.
7. **Transactional outbox** — `onAfterCommit` is fire-and-forget; crash-safe dispatch is needed before AI/Builder run dispatch can claim durability.
8. **Autopilot↔Cloud deploy bridge** — the first (Cloud) Deploy Target adapter: Cloud-side `/api/cloud/*` + a non-stub build executor; app-side Cloud client, Project↔cloud mapping, `cloud_deploy` action, and API key; orchestrated with `@questpie/workflows`.

The Builder *product* domain (Draft / Run / Checkpoint / Preview / Change-Request collections and lifecycle, provider-neutral git provider adapters at the product layer, Slovak-first meaning) stays in the app.

## Status and scope

This ADR is **accepted**. It fixes the architecture — one uniform sandbox contract across substrates, the app/framework split, the change-delivery and deploy-adapter shapes, and the credits tenancy model — but not the implementation, which is Phase 3/4. The Builder's editing surface and interaction flow (a possible IDE substrate with Autopilot as plugins, and its reconciliation with the Company Drive) are a deliberately separate later grill, recorded in the board note `builder-interaction-flow-and-ide-substrate-grill-mandate`.
