title: Authentication

Authentication is powered by [Better Auth](https://better-auth.com) with email/password, bearer tokens, and API keys. It shares the same SQLite database (`autopilot.db`) as the rest of the system.

## Authentication Is Always On

Authentication is enabled by default. You only need auth settings in `company.yaml` if you want to customize CORS or network controls:

```yaml title="company.yaml"
settings:
  auth:
    # cors_origin: "https://dashboard.example.com"  # optional, defaults to localhost dashboard origins
```

Then create the first owner account:

```bash title="terminal"
autopilot auth setup
# Prompts: name, email, password
# Saves a bearer token to ~/.autopilot/credentials.json
```

## CLI Commands

```bash title="terminal"
# Log in with email/password
autopilot auth login

# Log in with an API key (for CI/CD)
autopilot auth login --api-key ap_...

# Check current auth status
autopilot auth status

# Log out (deletes ~/.autopilot/credentials.json)
autopilot auth logout
```

CLI credentials are stored at `~/.autopilot/credentials.json` with file permissions 0600. The CLI sends either a `Bearer` token or `X-API-Key` header on every request.

## Actor Resolution

The auth middleware resolves every request to an Actor using this priority:

| Priority | Condition | Result |
|----------|-----------|--------|
| 1 | Path starts with `/hooks/*` | Webhook actor with member permissions |
| 2 | `X-API-Key` header present | Agent key or Better Auth API key |
| 3 | `Authorization: Bearer` header | Better Auth session (human), agent key fallback |
| 4 | No credentials | 401 Unauthorized |

### Actor Types

| Type | Source | Example |
|------|--------|---------|
| `human` | Dashboard login, CLI `auth login` | Owner, team members |
| `agent` | Agent API key (`ap_{agentId}_{random}`) | AI agents running tasks |
| `api` | Better Auth API key (X-API-Key) | CI/CD pipelines, external tools |
| `webhook` | `/hooks/*` paths | Linear, GitHub, Slack webhooks |

## Agent Keys

Agent API keys are generated fresh on every orchestrator startup. They are ephemeral — only SHA-256 hashes are persisted in `.auth/agent-keys.yaml` (file permissions 0600). The raw key exists only in memory during the process lifetime.

Key format: `ap_{agentId}_{base64url(24 random bytes)}`

## Roles & Permissions (RBAC)

Roles are defined in `team/roles.yaml`. Human-to-role mapping is in `team/humans.yaml`.

```yaml title="team/roles.yaml"
roles:
  owner:
    description: "Full access"
    permissions: ["*"]

  admin:
    description: "Manage users, approve tasks, view secrets"
    permissions:
      - tasks.*
      - agents.*
      - knowledge.*
      - chat.*
      - dashboard.*
      - secrets.read
      - team.read
      - team.invite
      - integrations.*
      - audit.read

  member:
    description: "Work on tasks, chat, read knowledge"
    permissions:
      - tasks.read
      - tasks.create
      - tasks.update
      - agents.read
      - chat.*
      - knowledge.read
      - knowledge.write
      - dashboard.read
      - team.read

  viewer:
    description: "Read-only access"
    permissions:
      - tasks.read
      - agents.read
      - dashboard.read
      - knowledge.read
      - team.read
```

### Team Members

```yaml title="team/humans.yaml"
humans:
  - email: founder@company.com
    role: owner
  - email: dev@company.com
    role: member
  - email: stakeholder@company.com
    role: viewer
```

Permission format is `resource.action`. Wildcard `*` grants all actions on a resource. The owner role has `["*"]` which grants everything.

### Permission Resources

| Resource | Actions |
|----------|---------|
| `tasks` | read, create, update, delete, approve, reject |
| `team` | read, invite, change_role, remove |
| `agents` | read, configure |
| `secrets` | read, create, delete |
| `knowledge` | read, write |
| `chat` | write |
| `integrations` | read, connect, disconnect |
| `audit` | read |
| `dashboard` | read |
| `sessions` | read, revoke |

## Two-Factor Authentication (2FA)

QUESTPIE Autopilot supports TOTP-based two-factor authentication via the Better Auth `twoFactor` plugin.

### Enabling 2FA

```bash title="terminal"
autopilot auth 2fa enable
# Displays an otpauth:// URI — paste into your authenticator app
# Shows 10 backup codes — save securely
# Prompts for verification code to activate
```

### CLI Commands

```bash title="terminal"
# Enable 2FA
autopilot auth 2fa enable

# Verify a TOTP code (e.g., after login)
autopilot auth 2fa verify

# Disable 2FA
autopilot auth 2fa disable
```

### Behavior

- **Human users:** When 2FA is enabled, sessions must be verified with a TOTP code before accessing API endpoints. Unverified sessions receive `401 Unauthorized`.
- **Agents:** API key authentication bypasses 2FA entirely — agents don't use sessions.
- **Trusted devices:** Verified devices are remembered for 30 days.
- **Backup codes:** 10 single-use backup codes are generated on enable. Use `POST /api/auth/two-factor/verify-backup-code` if you lose access to your authenticator.

## Session Management

Manage active sessions through the API or CLI.

### CLI Commands

```bash title="terminal"
# List active sessions
autopilot auth sessions

# Revoke a specific session
autopilot auth revoke <session-id>

# Revoke all sessions
autopilot auth revoke-all
```

### API Endpoints

| Method | Path | Description | Permission |
|--------|------|-------------|------------|
| GET | `/api/sessions` | List active sessions | `sessions.read` |
| DELETE | `/api/sessions/:id` | Revoke a session | `sessions.revoke` |
| DELETE | `/api/sessions` | Revoke all sessions | `sessions.revoke` |

## Cookie & Rate Limit Security

Better Auth enforces:

| Protection | Setting |
|-----------|---------|
| CSRF | `sameSite: strict` cookies |
| XSS | `httpOnly: true` (no JavaScript access to cookies) |
| HTTPS | `secure: true` when `NODE_ENV=production` |
| Login brute-force | 10 attempts per 5 minutes |
| Signup abuse | 5 attempts per 5 minutes |
| Global rate limit | 30 requests per minute |

## CORS

CORS origin is configurable via `settings.auth.cors_origin` in `company.yaml`. By default, local dashboard origins are allowed. For production, restrict to your dashboard domain:

```yaml title="company.yaml"
settings:
  auth:
    cors_origin: "https://dashboard.mycompany.com"
```

The `Authorization` and `X-API-Key` headers are explicitly allowed in CORS preflight responses.
