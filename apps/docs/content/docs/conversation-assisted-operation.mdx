---
title: Conversation-Assisted Operation
description: Bind external chat threads to tasks for bidirectional async operation.
---

Autopilot supports conversation bindings — connecting an external chat thread (Slack, Telegram, Discord, or any conversation provider) to a specific task. This lets operators take task actions and receive updates directly from their existing communication tools.

## How it works

1. A conversation binding is created, linking an external thread to a task
2. **Inbound**: when a message arrives on that thread, the conversation provider normalizes it into a task action
3. The orchestrator executes the action and the workflow continues
4. **Outbound**: when the bound task updates (blocked, failed, completed), the orchestrator delivers a notification back into the same thread

## Creating a binding

Bindings connect an external conversation identity to a task via the API:

```bash
# Local development (X-Local-Dev bypass — only works with `autopilot start`)
curl -X POST http://localhost:7778/api/conversations/bindings \
  -H "Content-Type: application/json" \
  -H "X-Local-Dev: true" \
  -d '{
    "provider_id": "text-conversation",
    "external_conversation_id": "chat-123",
    "external_thread_id": "thread-456",
    "mode": "task_thread",
    "task_id": "task-xxx"
  }'

# Production — use session auth or API key instead
curl -X POST https://your-orchestrator.example.com/api/conversations/bindings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{ ... }'
```

The binding stores:
- Which provider handles the conversation
- The external conversation/thread identity
- Which task is bound
- Binding mode (`task_thread` for task-scoped interaction)

## Inbound actions

When a message arrives on a bound thread, the handler normalizes it:

| User message | Normalized action | Effect |
|---|---|---|
| `/approve` | `task.approve` | Approves the blocked step |
| `/reject [reason]` | `task.reject` | Rejects the step with the message as reason |
| Any other text | `task.reply` | Feeds message into workflow as operator feedback |

The conversation provider handler decides how to interpret messages. The orchestrator trusts the normalized action.

## Outbound updates

When a bound task changes state, the orchestrator delivers updates back into the same conversation thread:

| Event | Delivered to thread |
|---|---|
| Task blocked (needs approval) | Title, task link, "needs attention" alert |
| Run completed | Summary, preview URL if available |
| Run failed | Error excerpt, run link |

Outbound delivery includes `conversation_id` and `thread_id` in the payload so the handler can post to the correct external thread.

## Auth

Inbound callbacks are protected by provider-secret auth. Each conversation provider must declare an `auth_secret` in its `secret_refs`. Callbacks must include the secret in the `X-Provider-Secret` header.

```yaml
# .autopilot/providers/my-conversation.yaml
id: my-conversation
kind: conversation_channel
handler: handlers/my-conversation.ts
capabilities:
  - op: conversation.ingest
  - op: notify.send
events:
  - types: [run_completed]
    statuses: [failed, completed]
  - types: [task_changed]
    statuses: [blocked]
secret_refs:
  - name: auth_secret
    source: env
    key: MY_CONVERSATION_SECRET
```

## What conversation bindings are not

- **Not a chat product.** There is no internal message store or timeline UI.
- **Not automatic.** Bindings must be explicitly created for specific tasks.
- **Not a general messaging system.** Bindings are task-scoped operational tools.

## The full async loop

With conversation bindings, the operator loop becomes:

1. Task is created (via CLI, intake, or conversation)
2. Workflow runs → reaches approval step → task blocks
3. Bound conversation thread receives "Task needs attention" notification
4. Operator types `/approve` or replies with feedback in the same thread
5. Workflow continues → run completes
6. Bound thread receives completion notification with preview URL
7. Operator opens preview from any device

This works without Worker App, web UI, or any custom client. The chat surface is the client.
