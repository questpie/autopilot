# Execution

## Running Tasks

```bash
# Run the next ready task
qap run-next

# Run a specific task
qap run-task QUE-238

# Run autonomous loop (all ready tasks sequentially)
qap run

# Dry-run (preview prompt, no side effects)
qap run-task QUE-238 --dry-run --no-sync
```

## Execution Loop

For each task, the runner follows this lifecycle:

1. **Find ready task** — compute from dependency graph, prioritize by track (gate > main > sidecar)
2. **Transition state** — `todo → ready → in_progress`
3. **Build steering** — load project steering, task notes, session notes
4. **Render prompt** — combine shared context + epic context + task metadata + task prompt + steering + mode instructions
5. **Spawn agent** — run Claude Code or Codex CLI with the prompt
6. **Capture result** — stdout, stderr, exit code, duration
7. **Transition** — `in_progress → implemented`
8. **Primary validation** — re-run agent with validation prompt
9. **Secondary validation** — another validation pass
10. **Validate command** — run `validateCommand` if configured (with warn/block policy)
11. **Finalize** — `committed → done`, recompute ready queue, emit events

If any step fails, the task transitions to `failed`. If `stopOnFailure` is set, the loop stops.

## Retry Policy

Tasks can define a retry policy:

```typescript
{
  id: "QUE-238",
  // ...
  retryPolicy: { maxAttempts: 3 },
}
```

On failure, the runner retries up to `maxAttempts` times before marking the task as failed.

## Validation

### Agent Validation

After implementation, two validation passes run using separate prompts:

1. **Primary** — checks acceptance criteria, types, tests
2. **Secondary** — checks regressions, contract compliance, scope creep

A validation pass is considered failed if the agent returns `Result: FAIL` in its output.

### Validate Command

If configured, a shell command runs after agent validation:

```typescript
execution: {
  validateAfterEachTask: true,
  validateCommand: "bun validate --filter questpie",
  validateCommandPolicy: "block",  // "warn" or "block"
}
```

- `"warn"` — log the failure, continue
- `"block"` — fail the task

## Session Recording

Every `run` / `run-next` / `run-task` creates a session record:

```json
{
  "id": "session-uuid",
  "projectId": "questpie-v3",
  "workspaceId": "users-dev-repos-cms",
  "startedAt": "2026-03-21T20:00:00Z",
  "status": "running",
  "provider": "claude",
  "taskCount": 60,
  "tasksCompleted": 3,
  "tasksFailed": 0,
  "currentTaskId": "QUE-238",
  "lastEventAt": "2026-03-21T20:05:00Z"
}
```

Sessions are updated **progressively** during execution. The TUI polls session state every 3 seconds for live progress.

## Dependency Resolution

A task is **ready** when all its `dependsOn` tasks are in a "done enough" state:

- For `gate` track dependencies: must be fully `done` or `committed`
- For `main`/`sidecar` dependencies: `validated_primary` or later is sufficient

`qap next` shows all ready tasks sorted by track priority.

`qap show <id>` shows what a task unlocks — which blocked tasks would become ready if this task completes.

## Prompt Assembly

The final prompt sent to the agent is assembled from multiple sources:

| Order | Section | Source |
|-------|---------|--------|
| 1 | Steering Notes | Project steering + task notes + session notes |
| 2 | Shared Context | `prompts.sharedContext` file |
| 3 | Epic Context | Epic ID, title, track + optional epic prompt file |
| 4 | Task Metadata | ID, title, kind, track, dependencies + states, acceptance criteria |
| 5 | Task Prompt | `task.promptFile` content |
| 6 | Source References | `task.sourceRefs` files (truncated to 5000 chars each) |
| 7 | Mode Instructions | Implementation or validation instructions |
| 8 | Output Format | Required output format template |

Sections are separated by `---` dividers.
