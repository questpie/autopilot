# Backlog Manifest

A `backlog.json` file is the primary input for `qap project import`. It defines the full task graph — real issue IDs, dependencies, epic groupings, and prompt file mappings.

## Why Backlog?

Without a backlog manifest, import relies on AI or generates generic `TASK-xxx` IDs. With a backlog:

- **Deterministic** — same input always produces the same config
- **Real issue IDs** — use your tracker IDs (`QUE-238`, `PROJ-42`, etc.)
- **Explicit dependencies** — you control the execution order
- **No AI required** — instant compilation, works offline

## File Location

Place `backlog.json` in the same directory as your prompt files:

```
my-prompts/
  backlog.json              # ← manifest
  00-shared-context.md      # shared context
  01-phase-1.md             # prompt for phase 1 tasks
  02-phase-2.md             # prompt for phase 2 tasks
  ...
```

The import command auto-detects it:

```bash
qap project import --name my-project --prompts ./my-prompts
```

Also detected: `project.manifest.json` (alternative name).

## Minimal Example

```json
{
  "version": 1,
  "project": {
    "id": "my-project",
    "name": "My Project"
  },
  "epics": [
    { "id": "EPIC-1", "title": "Core Feature", "track": "main" }
  ],
  "tasks": [
    {
      "id": "TASK-1",
      "title": "Set up schema",
      "epicId": "EPIC-1"
    }
  ]
}
```

## Field Reference

### Root Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `version` | `1` | Yes | Schema version. Must be `1`. |
| `project` | object | Yes | Project metadata. |
| `project.id` | string | Yes | Unique project identifier. |
| `project.name` | string | Yes | Human-readable project name. |
| `project.tracker` | object | No | Issue tracker integration. |
| `project.tracker.provider` | `"linear"` | Yes (if tracker) | Tracker provider. Currently only `"linear"`. |
| `project.tracker.projectId` | string | No | Tracker project ID. |
| `project.tracker.teamId` | string | No | Tracker team ID. |
| `sharedContext` | string | No | Filename of shared context file (relative to prompts dir). Loaded before every task prompt. |
| `epics` | array | Yes | At least one epic. |
| `tasks` | array | Yes | At least one task. |

### Epic Object

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Unique epic ID. Referenced by tasks via `epicId`. |
| `title` | string | Yes | Human-readable epic title. |
| `track` | `"main"` \| `"sidecar"` \| `"gate"` | Yes | Execution priority track. |
| `promptFile` | string | No | Epic-level prompt file (relative to prompts dir). |

### Task Object

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | string | Yes | — | Unique task ID (e.g., `QUE-238`). |
| `title` | string | Yes | — | Human-readable task title. |
| `epicId` | string | Yes | — | Must reference an existing epic ID. |
| `kind` | string | No | `"implementation"` | One of: `"implementation"`, `"validation"`, `"cleanup"`, `"migration"`, `"poc"`. |
| `track` | string | No | `"main"` | One of: `"main"`, `"sidecar"`, `"gate"`. |
| `dependsOn` | string[] | No | `[]` | Task IDs that must complete before this task can start. |
| `promptFile` | string | No | — | Task prompt file (relative to prompts dir). Multiple tasks can share the same prompt file. |
| `issueUrl` | string | No | — | URL to the issue in your tracker. |
| `acceptanceCriteria` | string[] | No | — | List of acceptance criteria shown in the task prompt. |

## Validation Rules

The manifest is validated on load. Import fails if any rule is violated:

1. **`version` must be `1`**
2. **At least one epic** and **at least one task** required
3. **Every task must reference an existing epic** via `epicId`
4. **No duplicate task IDs**
5. **All `dependsOn` references must exist** — no dangling references
6. **No dependency cycles** — detected via DFS traversal

## Tracks

Tracks control execution priority when multiple tasks are ready:

| Track | Priority | Use For |
|-------|----------|---------|
| `gate` | Highest | PoC tasks, validation gates that block phases |
| `main` | Normal | Core implementation work |
| `sidecar` | Lowest | Independent cleanup, docs, non-blocking work |

`qap next` returns gate tasks first, then main, then sidecar.

## Prompt File Mapping

The `promptFile` field maps tasks to prompt files in the prompts directory. Multiple tasks can reference the same prompt file (common for tasks within the same phase).

When a task runs, the prompt file content is loaded and injected into the agent prompt as the "Task-Specific Prompt" section.

If `sharedContext` is set at the root level, that file is loaded and injected as the "Shared Context" section before every task prompt.

## Creating from an Execution Plan

If you have an execution plan with issue tables, you can extract the backlog structure:

1. Each phase/group becomes an **epic**
2. Each issue row becomes a **task**
3. "blocked by" columns become **`dependsOn`** arrays
4. Gate/PoC issues get `"track": "gate"` and `"kind": "poc"`
5. Sidecar/parallel issues get `"track": "sidecar"`

The compilation is one-time. After import, you edit the compiled `autopilot.config.ts` if you need runtime adjustments.
